## 個人で作成する場合に簡単な点

- 共同作業の必要がないので、ブランチを切ったりマージしたりしなくても良い（してもいい）
  - コンフリクトすることがないので、何も考えずに main に push できる
- 自分の考えを人に伝えることなく、自分の頭の中でだけで作業ができる
- 開発のペースを自分で決められる
- コードスタイルや設計方針を統一する必要がない

## グループで作成する場合に工夫しないといけない点

- 記法を統一する
  - 誰がいつ読んでもすぐに理解しやすいようにする
  - コメントの書き方、変数名の命名規則など
- git flow に従ったブランチ設計を行う
  - ブランチ名を見ただけでどんな作業をしたのかがわかる
  - feature/機能名、bugfix/修正内容など
- 報連相を心がけて、作業分担を明確にする
  - 大きなコンフリクトを避けるため
  - 同じ作業を複数人で行わないようにする
- できるだけ差分が出ないようなコミットを心がける
  - レビュワーがレビューしやすいようにする
  - 1 つのコミットは 1 つの機能・修正に集中する
- Pull Request でのコードレビューを徹底する
  - バグの早期発見と知識共有のため
- 設計方針を統一する
	- DDD, TDD, MVC, MVVM, クリーンアーキテクチャ, オニオンアーキテクチャなど
